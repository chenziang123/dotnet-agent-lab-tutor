# .NET 实验助教 Agent — 团队分工说明

> **项目名称：** .NET 实验助教 Agent（答疑 + 轻量 RAG）  
> **团队人数：** 4 人  
> **开发周期：** 2 周  
> **协作模式：** 接力开发（A → B → C → D）  
> **最后更新：** 2026-06-08

---

## 一、项目概述

面向 .NET / AI Agent 课程实验的智能助教 Agent。学生用自然语言提问（实验步骤、概念理解、代码实现等），Agent 通过 **ReAct 推理循环** 自主规划，调用工具检索课程文档（轻量 RAG），并给出带引用来源的解答。

### 必做功能

| 要素 | 说明 |
|------|------|
| LLM 集成 | 通过 API 调用大语言模型 |
| Agent Loop | Thought → Action → Observation 迭代循环 |
| Tool Calling | 至少 3 个自定义工具/插件 |
| Memory | 对话记忆 + 工作记忆 |
| 用户界面 | Web（Blazor）或控制台 |

### 加分项目标

| 加分项 | 分值 | 负责人 |
|--------|------|--------|
| RAG / 向量检索 | +3 | B |
| Multi-Agent 协作 | +3 | D |
| 单元测试 | +2 | B（2 个）+ C（1–2 个） |
| MCP 服务器 | +2 | 不做（时间有限） |
| 创新性 | +2 | 实验助教场景 + 推理过程可视化 |

```
DotNetLabTutor/
```

## 三、接力开发流程

```
A 搭地基（3天） → B 做 RAG（3天） → C 接 Tool/Memory/Agent（3天） → D 做 Multi-Agent + UI + 交付（4天）
```

**接力原则：**

- 上一棒必须「能跑、有说明、有交接文档」，下一棒在其基础上扩展，不重写
- 每棒结束打 Git tag：`v0.1-a-done` → `v0.2-b-done` → `v0.3-c-done` → `v1.0-release`
- 每人提交 `docs/handoff-X.md` 交接文档

## 四、成员分工详情

### 成员 A — 项目地基 + Agent 骨架

**时间：** 第 1–3 天  
**工作量：** 约 25%

#### 任务清单

| 序号 | 任务 | 说明 |
|------|------|------|
| 1 | 创建 Solution | 4 个子项目：Core / Rag / Tools / Web |
| 2 | 基础设施 | DI 容器、`appsettings.json`、环境变量、日志（`ILogger`） |
| 3 | LLM 封装 | 统一 `ILlmService` / Chat Client，确保模型可调用 |
| 4 | Agent 骨架 | 最小 ReAct 循环，暂用 **Mock Tool** 验证 Thought → Action → Observation |
| 5 | 接口定义 | 定义 `IRagService`、`ISessionMemory` 供后续实现 |
| 6 | System Prompt | 助教人设、ReAct 格式、必须引用来源 |
| 7 | README 骨架 | 环境配置、构建与运行说明 |
| 8 | 交接文档 | 编写 `docs/handoff-A.md` |

#### 交付标准（交给 B 前必须满足）

- [ ] `dotnet run` 能启动，控制台可对话
- [ ] Agent Loop 能跑通 Thought → Action → Observation（Mock Tool 即可）
- [ ] `IRagService` 接口 + 空实现已就绪
- [ ] `docs/handoff-A.md` 已提交（项目结构、接口约定、运行方式）

#### 答辩职责

- 讲解：项目分层设计、ReAct 循环框架
- 准备 PPT 素材：Agent 架构图、ReAct 流程说明（第 12 天前交给 D）

### 成员 B — RAG 知识库

**时间：** 第 4–6 天  
**工作量：** 约 25%

#### 任务清单

| 序号 | 任务 | 说明 |
|------|------|------|
| 1 | 文档整理 | 整理 `resource/` 下 8–15 篇 Markdown 课件/实验说明 |
| 2 | 文档切块 | 实现 `DocumentChunker`，按 `##` 标题切分，每块 300–600 字 |
| 3 | Embedding | 接入 Embedding API，启动时写入内存向量库 |
| 4 | 检索服务 | 实现 `RagService : IRagService` |
| 5 | 核心 API | `SearchAsync(query, topK)` 返回 content + source + section + chunkId |
| 6 | 启动加载 | 启动时自动扫描 `resource/` 建索引 |
| 7 | 单元测试 | 2 个：切块逻辑、检索 Top-K |
| 8 | 交接文档 | 编写 `docs/handoff-B.md` |

#### 交付标准（交给 C 前必须满足）

- [ ] 独立调用 `SearchAsync("ReAct是什么")` 能返回相关文档片段
- [ ] 每条结果包含来源文件名和章节名
- [ ] 单元测试通过
- [ ] `docs/handoff-B.md` 已提交（切块策略、Embedding 用法、5 条测试 query）

#### 答辩职责

- 讲解：RAG 完整流程（切块 → Embedding → 检索 → 注入上下文）
- 准备 PPT 素材：RAG 流程图、切块策略说明（第 12 天前交给 D）

### 成员 C — Tools + Memory + Agent 联调

**时间：** 第 7–9 天  
**工作量：** 约 25%

#### 任务清单

| 序号 | 任务 | 说明 |

| 1 | Tool: SearchCourseDocs | 封装 B 的 RAG 检索，返回带引用 metadata |

| 6 | 对话记忆 | 多轮 messages 历史维护 |
| 7 | 工作记忆 | 当前实验编号、已检索文档、用户卡点 |
| 8 | Agent 联调 | 去掉 Mock Tool，接入真实 Tool，完善错误处理 |
| 9 | 推理日志 | 统一 `[AgentStep]` 前缀日志，供 UI 展示 |
| 10 | 单元测试 | 1–2 个 Tool 入参/边界测试 |

#### 交付标准（交给 D 前必须满足）

- [ ] 多轮追问可用（Memory 生效）
- [ ] 回答包含文档引用来源
- [ ] 检索为空时明确回复「知识库未找到」，不编造
- [ ] 准备好 5 条 Demo 问题及预期结果
- [ ] `docs/handoff-C.md` 已提交

#### 答辩职责

- 讲解：各 Tool 作用、Memory 设计、Tool 如何挂接到 Agent Loop

## 八、Git 协作规范

- **分支策略：** 每棒从上一棒 tag 拉 `feature/b-rag`、`feature/c-tools` 等
- **合并时机：** 每棒验收通过后合并到 `main` 并打 tag
- **禁止：** 直接在他人分支上改代码；未验收就交接
- **commit 格式：** `[A] 创建 Agent 骨架` / `[B] 实现 RAG 检索`

## 九、风险与缓冲

| 风险 | 应对 |
|------|------|
| B 的 RAG 延迟 | C 先用 B 的 `SearchAsync` 接口写 Tool，与 Agent 联调并行 |
| Embedding API 不可用 | 降级为关键词 + 全文搜索，报告中说明设计决策 |
| Demo 翻车 | 只用知识库内有的 5 个标准问题，禁止现场即兴提问 |
